StatTrack Database

Overview

This project is a culmination of the knowledge we have gained throughout this semester, as well as our attempt to include data structure and desing principles from previous courses. Our goal was to explore and implement various technologies/frameworks to expand our understanding of building and maintaining a fully fledged, operational application that utilizes database interactions.

We set out to design a web application that would allow users to favorite and track specific sports teams and players while simultaneously viewing aggregate historical statistic data. We were advised to limit our breadth of scope to just one particular sport and league i.e. NCAAMB, MLB, NFL. Our initial database logic design assumed we would be tracking historical and current data for NCAA football. As we began to delve further into the project, we soon discovered that this would not be possible due to time constraints and a lack of (free) readily accessible data. For this reason, we shifted our focus to tracking historical data of the MLB back to the mid 1800s.

We were able to find a github repository that contained excel sheets with all of the data that we hoped to incorporate. Unfortunately, the repository only contained seasonal data for each team and player and we were unable to include per-game statistic data (a link to the data repository can be found here). Cody Murrell wrote various VBA scripts that parsed the data files and inserted the data into MSSQL tables. Once the data was extracted, our database followed a similar pattern to the initial proposal. Once the database was initialized, we focused the implementation of various features including - user interface design, encrypted authentication, favoriting specific teams and players, sorting and displaying data, querying the Twitter API, and managing a user-driven forum.

Under the "My Feed" tab, the user will find an automatically generated Twitter feed based on their favorited teams and players. We query the Twitter API for the most recent data pertaining to the users preferences. Under the "Teams" tab, the user will find a list of teams either having been disbanded, or still participating in the MLB. If the user clicks on a team, they will be redirected to a new page with specific information on that team. Similarly, under the "Players" tab, users will find a list of all players who have been signed to a MLB team. If the user selects a player, they will be redirected to a page containing player specific and seasonal information. For both of the above options, the user has the option to "star" a player signifying that they would like to track that entity. They then have the ability to quickly navigate to favorited teams or players.

Technical Description

Below you will find a description of the system, including languages, tools, libraries and how each were used.

Languages

The entire front-end of the application was written using the Angular framework (v7). Angular is a TypeScript based front-end framework produced and maintained by Google. TypeScript, also referred to as ECMAScript6, is a superset of JavaScript, ECMAScript5. Angular also employs the use of template-binded HTML and CSS to develop web applications that are capable of running on various systems. According to the Angular documentation, "Angular combines declarative templates, dependency injection, end to end tooling, and integrated best practices to solve development challenges." Our application is currently being hosted on an AWS S3 bucket.

Furthermore, our middle-ground API was developed using the NestJS framework, a TypeScript based Node.js framework, similar to Angular. The purpose of the API was to bridge the connection gap between the front-end application and the back-end database. The API is currently being hosted on Heroku. The Angular application makes HTTP requests to the Heroku application which then queries the database.

Finally, our database was written using the Microsoft SQL database language. We used various VBA and shell scripts to extract data from excel sheets and initialize our database. The database is currently being hosted on Microsoft Azure with a backup server on AWS.

Tools

We primarily used various hosting services that could handle the "dev ops" of our app. At first, we attempted to use an AWS EC2 compute engine running Windows Server 2019 to host our database. After a slew of problems with the EC2 engine (and a lack of computational power), we decided it best to transfer our database over to Microsoft Azure, utilizing their free student tier access.
